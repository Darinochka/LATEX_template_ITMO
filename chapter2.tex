\section{Рабочая программа дисциплины}

Тематическое содержание разрабатываемого курса определяется выводами
раздела~\ref{sec:course_analysis}: классические алгоритмы машинного обучения
образуют обязательное ядро всех семи рассмотренных программ, а Jupyter
Notebook является стандартной средой для практических заданий. Соответственно,
в курсе этот контур воспроизводится: программа сосредоточена на классических
разделах ML, а домашние задания и семинарские задачи выполняются в Jupyter
Notebook --- среде, в которую впоследствии встраивается разрабатываемый
БЯМ-оценщик.

\subsection{Пререквизиты}

Для успешного освоения курса студент должен владеть:

\begin{itemize}
    \item основами программирования на Python (синтаксис, функции, классы, работа с библиотеками);
    \item линейной алгеброй (матрицы, векторы, операции над ними);
    \item математическим анализом (производные, градиент);
    \item основами теории вероятностей и математической статистики.
\end{itemize}

\subsection{Разделы}

Курс состоит из восьми тематических разделов, охватывающих основные направления классического машинного обучения:

\begin{enumerate}
    \item Введение в машинное обучение. Инструменты: NumPy, Pandas.
    \item Линейная регрессия и методы оптимизации.
    \item Классификация: логистическая регрессия, метод опорных векторов.
    \item Деревья решений.
    \item Ансамблевые методы: случайный лес, градиентный бустинг.
    \item Кластеризация.
    \item Специализированная тема по выбору: обработка естественного языка, компьютерное зрение или рекомендательные системы.
    \item Консультация по проекту и защита проектов.
\end{enumerate}

\subsection{Темы}

В рамках каждого раздела рассматриваются следующие темы:

\begin{itemize}
    \item Разведочный анализ данных, предобработка, feature engineering, работа с дисбалансом классов.
    \item Математические основы машинного обучения: функции потерь, методы оптимизации (градиентный спуск и его модификации: SGD, Momentum, Adam).
    \item Линейные модели: линейная и логистическая регрессия, метод наименьших квадратов.
    \item Метрики качества классификации и регрессии: Precision, Recall, F1-мера, ROC-AUC, PR-AUC, MSE, RMSE, MAE.
    \item Метод опорных векторов и ядровой трюк.
    \item Деревья решений для регрессии и классификации: критерии разбиения, переобучение, регуляризация.
    \item Ансамблевые методы: бэггинг, случайный лес, градиентный бустинг, bias-variance trade-off.
    \item Методы кластеризации и уменьшение размерности.
\end{itemize}

\subsection{Оцениваемые задания}

\subsubsection{Домашние задания}

На курсе предусмотрено три домашних задания, каждое рассчитано на две недели выполнения. Дедлайны строгие; работы, поданные после срока, не принимаются. Итоговая оценка за домашние задания --- среднее арифметическое трёх работ.

\begin{itemize}
    \item Домашнее задание 1. Работа с NumPy: векторизованные вычисления, операции над массивами.
    \item Домашнее задание 2. Градиентный спуск. Реализация с нуля классов VanillaGradientDescent, StochasticGradientDescent, MomentumDescent, Adam, а также класса LinearRegression с поддержкой L1/L2-регуляризации. Все реализации должны быть векторизованы. С использованием разработанных классов необходимо обучить модель, предсказывающую выбросы CO2. Датасет был самостоятельно сформирован из открытых источников. За прохождение определенных порогов целевой метрики студенты получали дополнительные баллы.
    \item Домашнее задание 3. Классификация текста. Предобработка текста, построение признаковых представлений, обучение и настройка классических моделей (логистическая регрессия, SVM), анализ метрик классификации и интерпретация результатов на задаче определения автора текста (человек или нейросеть). Домашнее задание частично выполняется в формате соревнования --- наивысшую оценку (10 баллов) могут получить только топ-10 студентов, получивших максимальное значение целевой метрики на валидационной выборке.
\end{itemize}

\subsubsection{Активность на семинарах}

В ходе каждого семинара студентам предлагаются бонусные задания и ответы на вопросы. За одно занятие можно получить не более четырёх отметок активности. Итоговая оценка за активность --- среднее за все занятия семестра. Шкала перевода:

\begin{itemize}
    \item 4 отметки --- 10 баллов;
    \item 3 отметки --- 7,5 балла;
    \item 2 отметки --- 5 баллов;
    \item 1 отметка --- 2,5 балла;
    \item 0 отметок --- 0 баллов.
\end{itemize}

Вопросы на семинарах формулировались так, чтобы подтолкнуть студентов к самостоятельному выводу, не давая ответ напрямую. Примеры вопросов: влияние коэффициента регуляризации $\lambda$ на поведение модели при его уменьшении или росте; причины отказа от масштабирования OneHot-закодированных признаков; интерпретация метрики $R^2$.

Часть заданий носила практический характер: студентам предлагалось написать код для обучения модели или провести эксперименты с гиперпараметрами --- например, исследовать влияние параметров случайного леса на качество модели и OOB-score.

\subsubsection{Проект}\label{sec:project}

Проект выполняется индивидуально (командные работы допускаются в исключительных случаях по согласованию с преподавателем). Работа ведётся в репозитории GitHub Classroom и сдаётся поэтапно через четыре чекпоинта с последующей защитой.

Требования к проекту:
\begin{itemize}
    \item датасет --- не менее 10\,000 строк и не менее 9 признаков;
    \item наличие чётко сформулированной задачи машинного обучения;
    \item уникальность: датасет не должен пересекаться с проектами других студентов;
    \item при использовании Kaggle --- только задачи типа monetary, задачи Getting Started не допускаются.
\end{itemize}

Структура сдачи и максимальные баллы:

\begin{itemize}
    \item CP0 (выбор проекта, регистрация в GitHub Classroom) --- допускной этап;
    \item CP1 (предобработка данных, базовое моделирование, качество кода) --- 25 баллов, проходной порог 13 баллов;
    \item CP2 (расширенное моделирование, перебор гиперпараметров, ансамбли) --- 30 баллов, проходной порог 15 баллов;
    \item CP3 (отчёт, деплой через FastAPI или аналог) --- 30 баллов, проходной порог 15 баллов;
    \item защита (презентация 5 минут, ответы на вопросы) --- 15 баллов.
\end{itemize}

Не набравший проходной балл на текущем чекпоинте не допускается к следующему; итоговая оценка за проект формируется из уже накопленных баллов.

\section{Дизайн эксперимента по оценке БЯМ-оценщика}\label{sec:eval_design}

Прежде чем встраивать БЯМ-оценщик в инфраструктуру проверки заданий, необходимо
эмпирически подтвердить, что выбранный подход действительно обеспечивает
качество оценки, сопоставимое с экспертным. Это требование непосредственно
вытекает из гипотезы, сформулированной в разделе~\ref{sec:llm_judge}:
автоматическая оценка должна быть согласованной с ручной проверкой
преподавателя без дополнительных временных затрат с его стороны.

\subsection{Объект оценивания и подход}

Объектом оценки является БЯМ-оценщик, на вход которому подаётся вопрос,
ответ студента и рубрика, а на выходе ожидается балл по дискретной шкале.
Естественной единицей анализа здесь служит одна пара <<вопрос--ответ>>: на
каждой такой паре оценщик выставляет ровно один балл, который сопоставляется
с эталонным.

Для оценивания ответов студентов выбрана дискретная шкала из четырёх уровней:
0, 1, 2, 3 балла. Нулевой балл соответствует пустому или полностью
нерелевантному ответу; баллы 1, 2 и 3 описываются индивидуальными критериями
для каждого вопроса. Такая шкала является достаточно подробной, чтобы отражать
качественные различия между ответами разной полноты, и одновременно достаточно
компактной, чтобы критерии можно было сформулировать однозначно.

Анализ подходов к LLM-as-a-Judge показал, что качество БЯМ-оценщика существенно
зависит от модели, формата промпта и наличия структурированных критериев
оценивания. Поэтому ключевая исследовательская задача настоящей главы ---
подобрать сочетание модели и промпта, обеспечивающее приемлемое качество
оценивания на типах заданий, характерных для курса по классическому машинному
обучению. Для этого необходимо: зафиксировать метрику качества,
определить охват предметной области и типов вопросов, собрать датасет
вопросов с эталонными ответами и рубриками, сгенерировать на этой основе
модельные ответы студентов с заранее известными баллами и провести серию
экспериментов с различными конфигурациями оценщика.

\subsection{Метрика качества}

В качестве метрики качества согласованности БЯМ-оценщика с экспертной оценкой
используется доля точного совпадения (exact match, accuracy) ---
доля случаев, когда автоматически выставленный балл совпадает с эталонным.
Выбор этой метрики обусловлен соображениями интерпретируемости: на дискретной
шкале из четырёх значений доля точного совпадения непосредственно отражает,
как часто оценщик ставит ровно тот балл, что и эксперт, и не требует
дополнительного пояснения, в отличие от Quadratic Weighted Kappa или
коэффициентов корреляции (раздел~\ref{sec:llm_judge}).

\subsection{Формирование датасета}\label{sec:dataset}

\subsubsection{Генерация вопросов и эталонных ответов}

Ручная подготовка 105 разнообразных вопросов с эталонными ответами и
покритериальными рубриками является трудозатратной и плохо масштабируемой
процедурой. Поэтому первичная генерация делегирована БЯМ, а контроль качества
выполнен преподавателем вручную.

Чтобы снизить риск систематических смещений, обусловленных особенностями
конкретной модели --- в том числе self-enhancement bias, описанного в
разделе~\ref{sec:llm_judge}, --- генерация выполнялась шестью различными
моделями: Claude Sonnet 4.5, Claude Sonnet 4.6, Claude Haiku 4.5, Gemma~3,
Qwen~3 и GLM-5 turbo. Использование разных семейств моделей обеспечивает
стилистическое и содержательное разнообразие формулировок.

Каждой модели передавался единый системный промпт с требованиями к вопросу,
эталонному ответу и рубрике, а также пользовательский промпт с указанием
темы, типа вопроса и количества генерируемых единиц. Полные тексты системного и пользовательского промптов, использованных
при генерации, приведены в приложении~\ref{app:prompts}, разделе~\ref{app:prompts:generation}.
Системный промпт фиксирует следующие требования:

\begin{itemize}
    \item вопрос требует развёрнутого текстового ответа, не сводится к тесту
    или коду, конкретен и однозначен;
    \item уровень сложности соответствует студенту третьего курса
    технического вуза; вопросы не касаются базовых определений, а нацелены
    на более комплексные и неочевидные аспекты;
    \item эталонный ответ занимает 3--6 предложений, содержит ключевые
    понятия и при необходимости формулы в LaTeX-нотации, не содержит
    избыточной информации;
    \item рубрика состоит ровно из трёх описаний --- по одному на каждый
    из ненулевых баллов 1, 2, 3; формулировки критериев конкретны
    (<<студент указал, что~...>>, а не <<правильность ответа>>);
    \item ответ возвращается строго в формате JSON.
\end{itemize}

В результате каждая из шести моделей сгенерировала по 35 вопросов
с ответами и рубриками --- всего 210 вопросов.

\subsubsection{Ручной отбор и валидация}

Сгенерированный массив прошел ручную валидацию преподавателем. На этом
этапе исключались дубликаты и близкие переформулировки между моделями,
вопросы с фактическими ошибками, чрезмерно простые или, наоборот, выходящие
за рамки программы курса формулировки, а также рубрики, в которых критерии
для разных баллов были сформулированы неразличимо. По итогам отбора в
финальный датасет вошло 105 вопросов, равномерно покрывающих 35 пар
<<тема--тип>> по три вопроса на пару.

Результирующий датасет служит основой для последующих этапов: на нём
генерируются ответы студентов разного качества и проводится оценка
БЯМ-оценщика.

\subsubsection{Генерация ответов студентов разного качества}

Для оценки качества БЯМ-оценщика на дискретной шкале необходимо иметь набор
ответов, для каждого из которых известен <<правильный>> балл с точки зрения
рубрики. Сбор реальных ответов студентов на этом этапе невозможен (курс ещё
не запущен), а ручное написание ответов по 105 вопросам на каждый из
ненулевых баллов трудозатратно. Поэтому ответы генерируются БЯМ, после чего
валидируются преподавателем.

Для генерации использовалась открытая модель gpt-oss-120b. Модели передавался специализированный промпт, 
инструктирующий её отвечать на вопрос ровно на заданный балл: с использованием формулировки
соответствующего критерия рубрики и без добавления деталей, тянущих ответ
на более высокий уровень. 

Дополнительно требовалось писать в стиле,
характерном для студента, --- без избыточной формальности, а для низких
баллов --- максимально просто и кратко. Для каждого из 105 вопросов
сгенерировано четыре ответа: на 1, 2 и 3 балла --- моделью; на 0 баллов ---
пустой ответ, имитирующий ситуацию, когда студент не ответил вовсе.
Полные тексты промптов, использованных для генерации ответов студентов
заданного качества, приведены в приложении~\ref{app:prompts}, разделе~\ref{app:prompts:answers}.

После генерации все ответы прошли ручную валидацию: было проверено, 
действительно ли ответ соответствует целевому критерию и
заслуживает заявленного балла, и при необходимости корректировал
формулировки. Итоговый объём датасета --- 420 пар <<вопрос--ответ>> с
известными эталонными баллами, равномерно распределёнными по шкале
0--3 балла.

\subsection{Эксперименты}\label{sec:eval_experiments}

Собранный датасет из 420 пар <<вопрос--ответ>> с эталонными баллами
используется для сравнения конфигураций БЯМ-оценщика.
Цель экспериментов --- выбрать сочетание модели и промпта, обеспечивающее
наилучшее значение accuracy при приемлемых стоимости и скорости работы.

\subsubsection{Варьируемые параметры}

В рамках настоящей работы варьируются два основных параметра конфигурации
оценщика.

\textbf{Модель-оценщик.} В соответствии с анализом ландшафта БЯМ
(раздел~\ref{sec:llm_judge}) рассматриваются как проприетарные, так и
открытые модели разных уровней качества и стоимости. Для всех моделей используется единый
OpenAI-совместимый клиент через агрегатор, что позволяет менять модель
изменением одного параметра конфигурации.

\textbf{Шаблон промпта.} Сравниваются варианты промпта, различающиеся
наличием или отсутствием следующих элементов:
эталонного ответа во входных данных оценщика;
явного запроса цепочки рассуждений (Chain-of-Thought) перед выставлением
балла; указаний по митигации смещений (verbosity bias, self-enhancement
bias). Базовая часть промпта --- формулировка задания, ответ студента,
рубрика и максимальный балл --- остаётся неизменной во всех
конфигурациях.

\subsubsection{Протокол эксперимента}

Каждая конфигурация запускается на полном датасете из 420 пар. Для каждой
пары оценщик получает на вход вопрос, ответ студента, рубрику и --- в
зависимости от конфигурации --- эталонный ответ; на выходе ожидается
число от 0 до 3. Качество конфигурации
характеризуется значением accuracy --- долей пар, на которых полученный
балл совпадает с эталонным; параллельно фиксируются среднее время отклика
и суммарная стоимость прогона.

\section{Дизайн эксперимента по оценке агентной системы проверки
проектов}\label{sec:agent_eval_design}
 
Помимо БЯМ-оценщика теоретических ответов, разрабатываемая
инфраструктура включает многоагентную систему проверки проектных
репозиториев (раздел~\ref{sec:project_grading}). Поскольку входные
данные, шкала и архитектура у этой системы принципиально отличаются от
БЯМ-оценщика для коротких ответов, для неё формулируется собственный дизайн
эксперимента, описанный в настоящем разделе.
 
\subsection{Объект оценивания и подход}
 
Объектом оценки является система, на вход которой подаётся репозиторий
студента и описание чекпоинта, а на выходе формируются: численные баллы
и текстовый фидбек
по каждому критерию рубрики и агрегированные баллы по категориям. Оценка по каждой категории
выставляется отдельным БЯМ-агентом, который самостоятельно навигирует
по репозиторию через набор инструментов чтения файлов.
 
В отличие от БЯМ-оценщика, для которого эталонной единицей сравнения служит
балл на одну пару <<вопрос--ответ>>, у агентной системы естественной
единицей анализа является репозиторий целиком. Поэтому ключевые
метрики качества определяются в первую очередь на уровне репозитория и
категории, а не отдельного критерия.

Ключевая исследовательская задача --- проверить, что агентная система
способна выставлять баллы и формировать текстовый фидбек, согласованные
с экспертной разметкой, при приемлемой операционной надёжности и
стоимости. Для этого необходимо: зафиксировать набор метрик качества,
собрать выборку реальных репозиториев с эталонной разметкой, прогнать
систему в штатной конфигурации и измерить её поведение по выбранным
осям.

\subsection{Метрика качества}
 
Качество агентной системы оценивается по нескольким независимым
осям, каждая из которых отвечает за свой аспект практической
применимости.
 
Основная метрика ---
согласованность баллов системы с эталонными баллами. На каждой
категории отдельно вычисляются средняя абсолютная ошибка --- 
интерпретируется как среднее отклонение балла системы от балла
эксперта в шкале категории.

 
% \textbf{Устойчивость к стохастичности БЯМ.} БЯМ-агенты по своей природе
% недетерминированы, поэтому отдельной метрикой выступает разброс
% оценок при повторных прогонах на одном и том же репозитории. Из общей
% выборки выбираются 5 репозиториев, на каждом из которых система
% запускается 5 раз. Метрика --- среднее по репозиториям стандартное
% отклонение суммарного балла. Высокий разброс при низкой средней ошибке
% означает, что система согласуется с экспертом <<в среднем>>, но
% непригодна для однократного применения в реальной проверке, что
% является самостоятельной проблемой.
 
Помимо численной оценки система
формирует текстовый фидбек для студента, и его качество критично для
практического использования. Фидбек оценивается по двум независимым
осям:
\begin{itemize}
    \item соответствие заданному формату (бинарная метрика
    0/1) --- структура и форма фидбека, соответствие заданному промпту;
    \item фактическая корректность утверждений (трёхуровневая шкала
    0/1/2) --- доля содержательных утверждений в фидбеке, которые
    действительно соответствуют коду репозитория, в противовес
    галлюцинациям (упоминание несуществующих файлов или конструкций,
    приписывание свойств, которых в коде нет).
\end{itemize}
 
\textbf{Операционная надёжность и затраты.} Для оценки практической
применимости фиксируются:
\begin{itemize}
    \item доля успешных прогонов --- запусков, в которых все агенты
    завершились без сбоев, превышения таймаутов и зацикливания на
    повторных вызовах инструментов;
    \item среднее время полной проверки одного репозитория;
    \item средняя стоимость прогона по тарифам используемого
    агрегатора БЯМ;
    \item среднее число вызовов инструментов на одного агента ---
    показатель эффективности навигации по репозиторию.
\end{itemize}

\subsection{Формирование датасета}
 
В качестве бенчмарка используется выборка из 20 репозиториев студентов,
сданных по чекпоинту CP1 курса (раздел~\ref{sec:project}).
Чекпоинт CP1 выбран как наиболее показательный из доступных: он
охватывает все основные категории рубрики --- предобработку данных,
базовое моделирование и качество кода --- и формирует достаточный
объём содержательной работы для оценивания.
 
Для каждого репозитория из выборки выставляются численный балл по каждому критерию чекпоинта.
Далее эти баллы агрегируются в суммарный балл по каждой категории (сумма баллов по критериям данной категории) 
и суммарный балл за чекпоинт.

\subsection{Эксперименты}

Собранная выборка из 20 репозиториев с эталонной разметкой используется
для оценки качества агентной системы в штатной конфигурации. Цель
экспериментов --- измерить значения выбранных метрик и зафиксировать,
насколько система пригодна к практическому использованию в учебном процессе.

\subsubsection{Протокол эксперимента}

Эксперименты проводятся в следующем порядке:
\begin{enumerate}
    \item Размечается выборка из 20 репозиториев согласно
    рубрике CP1; разметка фиксируется и далее не изменяется.
    \item Каждый репозиторий проверяется агентной системой в
    штатной конфигурации; сохраняются итоговые JSON-оценки и
    сгенерированные файлы фидбека.
    \item Вручную размечаются фидбеки всех прогонов из основной
    выборки по двум осям (формат, фактическая корректность).
    \item Вычисляются метрики качества и фиксируются в сводной
    таблице.
\end{enumerate}
