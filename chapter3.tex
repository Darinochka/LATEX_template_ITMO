\section{Архитектура автоматизированной проверки в nbgrader}

В стандартной версии nbgrader для ячеек с кодом доступны следующие типы: Autograded answer - ячейка, в которой студент пишет код, проверяемый автоматическими assert-тестами; Autograded tests - ячейка с тестами, скрытая от студента и запускаемая при проверке; Manually graded answer - ячейка с кодом или текстом, требующая ручной оценки; Manually graded task - аналогичный тип, но оценка выставляется на уровне задания целиком; Read-only - ячейка, недоступная для редактирования студентом. Для текстовых (Markdown) ячеек набор типов ограничен: Manually graded answer, Manually graded task и Read-only. Таким образом, автоматическая проверка предусмотрена только для кода, а текстовые ответы могут быть оценены исключительно вручную.

\subsection{Требования к работе}

На основании проведенного анализа были сформулированы следующие требования к разрабатываемому решению.

Требования к новому типу ячейки:

\begin{enumerate}
    \item В nbgrader должен быть добавлен новый тип ячейки - LLM graded answer, доступный для выбора наряду со стандартными типами.
    \item Ячейка должна поддерживать структурированную разметку с разделителями: блок с формулировкой задания (видимый студенту) и блок с эталонным ответом и критериями оценивания (скрытый от студента).
    \item Разделители должны быть настраиваемыми через конфигурацию.
\end{enumerate}

Требования к генерации студенческой версии:

\begin{enumerate}
    \item При генерации студенческой версии задания блок с критериями должен автоматически удаляться и заменяться текстом-заглушкой.
    \item Формулировка задания должна сохраняться в неизменном виде.
    \item Генерация должна выполняться стандартной командой nbgrader без дополнительных утилит.
\end{enumerate}

Требования к конфигурации и промпту:

\begin{enumerate}
    \item Преподаватель должен иметь возможность задать параметры подключения к API языковой модели: ключ, адрес, название модели.
    \item Должен поддерживаться настраиваемый шаблон промпта с подстановкой переменных: формулировка задания, ответ студента, вывод ячейки, критерии оценивания, максимальный балл.
    \item Должен быть задан таймаут ожидания ответа от API.
\end{enumerate}

Требования к выходу модели и интеграции:

\begin{enumerate}
    \item Модель должна возвращать численную оценку в диапазоне от 0 до максимального балла за задание.
    \item Проверка ячеек типа LLM graded answer должна запускаться стандартной командой nbgrader autograde совместно с проверкой остальных типов ячеек.
    \item Результаты проверки должны сохраняться в базу данных nbgrader, обеспечивая совместимость с существующим интерфейсом просмотра оценок.
\end{enumerate}

\subsection{Конфигурация и запуск проверки}

Перед запуском автоматической проверки преподаватель заполняет конфигурационный файл, задающий параметры обращения к БЯМ. Ключевые поля конфигурации:


\begin{itemize}
    \item llm\_api\_key, llm\_api\_base, llm\_model - параметры подключения к API языковой модели;
    \item БЯМ\_prompt\_template - шаблон промпта, в который подставляются:
        \begin{itemize}
            \item \{question\} - формулировка задания;
            \item \{student\_answer\} - ответ студента;
            \item \{cell\_output\} - вывод ячейки (если есть);
            \item \{grading\_instructions\} - критерии оценивания;
            \item \{max\_points\} - максимальный балл за задание.
        \end{itemize}
    \item begin\_question\_delimiter, end\_question\_delimiter и аналогичные поля для критериев - настройка разделителей разметки; по умолчанию равны BEGIN QUESTION\_LLM и END QUESTION\_LLM для заданий и BEGIN CRITERIA\_LLM и END CRITERIA\_LLM для критериев.
    \item llm\_timeout - максимальное время ожидания ответа от API (в секундах).
\end{itemize}
Модель получает на вход перечисленные данные и возвращает число от 0 до max\_points, представляющее выставленный балл.

После получения ответов студентов преподаватель запускает проверку стандартной командой: nbgrader autograde.

Система автоматически обходит все ячейки типа LLM graded answer, отправляет соответствующие запросы к языковой модели и сохраняет выставленные баллы в базу данных nbgrader. Остальные типы ячеек проверяются штатными механизмами инструмента.

\subsection{Внедрение в курс}

Автоматическая проверка была внедрена во все домашние задания курса. Задания публиковались через LMS и размещались в репозиториях GitHub; студенты сдавали решения в виде файлов через ту же платформу. Для автоматизации процесса были разработаны скрипты, обеспечивающие загрузку файлов и запуск проверки.

Текущая система наследует ряд ограничений, присущих nbgrader:

\begin{itemize}
    \item \textbf{Удаление проверяющих ячеек.} Если студент удалял ячейку с тестами и заново формулировал ответ, nbgrader не распознавал его: каждая ячейка содержит уникальный идентификатор, по которому система отслеживает ответы и их результаты. При отсутствии ячейки задание считается невыполненным.

    \item \textbf{«Сломанные» ноутбуки.} В ряде случаев студенты сдавали файлы, которые nbgrader не мог обработать. Точные причины этого явления установить не удалось.
\end{itemize}

В подобных ситуациях домашние задания проверялись вручную. Доля таких случаев составила~5{,}0\% от общего числа студентов.

\section{Архитектура агентной системы}\label{sec:agent_architecture}

В отличие от расширения nbgrader, ориентированного на проверку коротких
теоретических ответов внутри отдельных ячеек ноутбука, проверка
семестровых проектов требует анализа репозитория целиком: совокупности
ноутбуков, скриптов, файлов конфигурации и сопутствующих артефактов.
Объём и разнородность входных данных не позволяют свести задачу к
одному вызову БЯМ с фиксированным промптом. В соответствии с дизайном,
зафиксированным в разделе~\ref{sec:agent_eval_design}, эта задача
решается многоагентной системой, в которой каждый агент специализирован
на своей категории рубрики и самостоятельно навигирует по репозиторию
через ограниченный набор инструментов. В настоящем разделе описана
архитектура этой системы на уровне основных компонентов и их
взаимодействия.

\subsection{Принципы построения и доменная модель}

Система спроектирована вокруг трёх принципов: отделение содержания
рубрик от программного кода, специализация агентов по категориям
рубрики и ограниченный доступ к репозиторию через явный набор
инструментов. Содержание рубрик и тексты промптов хранятся как
конфигурация, а ядро системы --- оркестратор, агенты и инструменты ---
не зависит от конкретного курса. Это позволяет адаптировать систему
под другую дисциплину без изменения исходного кода: достаточно
переопределить рубрику и промпты агентов.

В основе системы лежит доменная модель, описывающая предметную
область проверки:

\begin{itemize}
    \item \textbf{Категория} --- группа семантически близких критериев
    рубрики (например, <<предобработка данных>>); характеризуется
    идентификатором, описанием и предельной суммой баллов.
    \item \textbf{Критерий} --- атомарная единица оценивания; привязан
    к категории, имеет описание и максимальный балл.
    \item \textbf{Чекпоинт} --- именованный этап контроля (CP1, CP2);
    содержит список критериев, подлежащих проверке на данном этапе.
    \item \textbf{Вывод агента} --- результат работы одного агента над
    одной категорией: баллы по критериям с текстовым фидбеком и
    диагностические данные о ходе работы агента.
    \item \textbf{Результат проверки} --- итог проверки одного
    репозитория по одному чекпоинту: статус выполнения, выводы
    отдельных агентов, агрегированные баллы и сгенерированный фидбек
    для студента.
\end{itemize}

Над доменной моделью построен слой управления, реализующий операции
создания, изменения и удаления категорий, критериев и чекпоинтов.
Этот слой используется одинаково из всех пользовательских
интерфейсов и поддерживает инвариант <<одна категория --- один
агент>>: при создании новой категории автоматически появляется
соответствующий ей агент, при удалении категории --- удаляется и
агент.

\subsection{Оркестратор и слой агентов}

\textbf{Оркестратор пайплайна оценки.} Точкой входа в систему служит
оркестратор, выполняющий следующий сценарий проверки одного
репозитория по одному чекпоинту: загрузка определения чекпоинта и
разрешение списка критериев; подготовка локальной копии репозитория,
включая предобработку (очистку выходов ячеек ноутбуков, устраняющую
шум во входных данных и снижающую расход токенов); подбор агентов,
отвечающих за категории, представленные в чекпоинте; параллельный
запуск отобранных агентов; агрегирование баллов по категориям и
формирование итогового результата; персистентное сохранение
результата.

\textbf{Реестр агентов.} Агенты не задаются в коде, а описываются
декларативно --- через конфигурацию рубрики и тексты системных
промптов. Реестр предоставляет оркестратору операцию подбора
агентов под конкретный чекпоинт: возвращаются только те агенты, чьи
категории фактически встречаются в списке критериев чекпоинта,
и каждому агенту на исполнение передаются только критерии его
категории. Тем самым агент изолирован от посторонних критериев, а
его промпт не разрастается с увеличением рубрики.

\textbf{Агент категории.} Один экземпляр агента работает над одной
категорией рубрики. Внутренний цикл агента построен по
следующей схеме: модель чередует шаги рассуждения и вызовы
инструментов до тех пор, пока не сформирует итоговый ответ.
На вход агенту подаются системный промпт его категории, путь к
локальной копии репозитория, список критериев категории с
максимальными баллами и набор инструментов; на выходе ожидается
структурированный ответ с покритериальными баллами и фидбеком.
Число шагов рассуждения ограничено сверху, что предотвращает
бесконечные циклы. При нарушении формата ответа по всем критериям
выставляются нулевые баллы и сохраняется диагностическое сообщение
--- сбой одного агента не приводит к падению пайплайна.

\textbf{Инструменты для работы с репозиторием.} Все агенты
используют общий набор инструментов, описанных в
разделе~\ref{sec:agent_eval_design}: обход дерева файлов, чтение
файла или его фрагмента, поиск паттерна, чтение Jupyter-ноутбуков.
Инструменты ограничены параметрами безопасности --- предельным
временем исполнения, объёмом считываемых данных, длиной вывода
ноутбука. Это исключает ситуации, когда агент пытается прочитать
файл произвольного размера или зависает на обходе большой
директории. Каждый вызов инструмента фиксируется в выводе агента,
что обеспечивает наблюдаемость поведения и используется при отладке
промптов.

\textbf{Параллельный запуск агентов.} Отобранные агенты исполняются
конкурентно с ограничением степени параллелизма. Ошибка одного
агента локализуется: проблемная категория получает нулевые баллы с
текстом ошибки в фидбеке, остальные агенты продолжают работу штатно,
а итоговый результат сохраняется в полном объёме с явным указанием,
какая категория не была оценена. Такая схема обеспечивает
устойчивость пайплайна к точечным сбоям БЯМ и сетевых вызовов.

\subsection{Хранение результатов и формирование фидбека}

Результаты каждого прогона сохраняются изолированно: для каждой
проверки создаётся отдельная запись, идентифицируемая репозиторием,
чекпоинтом и временем прогона. Такая схема исключает коллизии при
повторных проверках одного и того же репозитория и упрощает
последующий массовый анализ результатов.

Помимо численных баллов система формирует текстовый фидбек для
студента. Фидбек собирается шаблонизатором из выводов отдельных
агентов; шаблон задаёт структуру фидбека.
Это позволяет адаптировать форму фидбека под стиль конкретного
преподавателя без изменения логики системы.

\subsection{Свойства полученной архитектуры}

В получившейся системе явно реализованы следующие инженерные
свойства, важные для применения в учебном процессе:

\begin{itemize}
    \item \textbf{Разделение содержания и поведения.} Предметная
    область описана как конфигурация, поведение агентов задано
    промптами, код оркестрации и инструментов отделён от содержания
    рубрик. Это позволяет переиспользовать систему для других
    курсов.

    \item \textbf{Масштабирование по категориям.} Один
    специализированный агент на категорию рубрики; параллельный
    запуск с ограничением конкурентности позволяет ускорять
    проверку без перегрузки БЯМ.

    \item \textbf{Безопасный и предсказуемый доступ к репозиторию.}
    Агент взаимодействует с файловой системой только через явно
    объявленные инструменты с таймаутами и лимитами объёма чтения,
    что исключает зависания и неконтролируемое потребление
    ресурсов.

    \item \textbf{Наблюдаемость.} Все вызовы инструментов и шаги
    цикла рассуждений модели фиксируются в структуре результата,
    что обеспечивает воспроизводимость оценки и упрощает отладку
    промптов.

    \item \textbf{Локализация сбоев.} Ошибка одного агента не
    приводит к потере результатов остальных: проблемная категория
    получает нулевые баллы с явным сообщением, остальные ---
    штатно оцениваются.
\end{itemize}

Полученная архитектура соответствует требованиям дизайна
эксперимента, сформулированным в разделе~\ref{sec:agent_eval_design},
и используется в качестве рабочей конфигурации при экспериментальной
оценке системы.

\section{Результаты апробации}

Разработанные инструменты были апробированы в рамках реального курса в НИУ ВШЭ. В апробации приняли участие 200 студентов третьего курса. В ходе курса фиксировалось среднее время от момента сдачи работы до получения студентом обратной связи~--- один из ключевых показателей эффективности системы проверки.
 
Результаты по каждому типу заданий приведены в таблице~\ref{tab:time_results}. Время указано в календарных днях --- от момента дедлайна сдачи до момента, когда студент получил фидбек; само время выполнения автоматической проверки в этот показатель не входит. Ключевое преимущество инструмента состоит в том, что проверка запускается преподавателем в фоновом режиме параллельно с другими задачами и не требует ручного разбора каждой работы, что в итоге сокращает организационную задержку и позволяет быстрее донести результат до студентов.
 
Для первого и третьего домашних заданий время составило 3 и 5 дней соответственно. Значительное отклонение для второго задания (11 дней) объясняется организационными причинами, не связанными с работой системы. Для первого чекпоинта проекта студенты получали оценки постепенно по мере обработки репозиториев --- от 1 до 3 дней. Данные по чекпоинтам CP2 и CP3 на момент написания работы ещё не собраны.
